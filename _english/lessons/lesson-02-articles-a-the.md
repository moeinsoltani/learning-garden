# Lesson 02: Articles I — a/an vs the

## Concept

Articles (*a*, *an*, *the*) are one of the hardest things in English for speakers
of languages that don't have them — and one of the most common error sources. The
good news: for choosing between *a/an* and *the*, there's **one question** that
handles most cases:

```
   "Does the reader already know WHICH one I mean?"

   NO  → use  a / an   (introducing it — one of many, not yet specified)
   YES → use  the      (we both know exactly which one)

   "I found a bug."          ← a bug: new, one of many, reader doesn't know which
   "I fixed the bug."        ← the bug: we both know which one (just mentioned it)
```

Think of it as *new* vs *known*. When you introduce something for the first time,
or it's just "one of several," the reader doesn't yet know which one you mean — use
**a/an**. Once it's been mentioned, or it's clear from context which specific one,
the reader knows which — use **the**.

---

## Going Deeper

### First mention *a*, later mentions *the*

The most reliable pattern: the first time you mention something, use *a/an*
(introducing it); after that, use *the* (now the reader knows which one).

- "I opened **a** pull request. **The** PR fixes the login bug." (First *a PR* —
  new; then *the PR* — now you know which one.)
- "We hit **an** error in production. **The** error only happens under load."

### *the* for shared context

Use *the* even on first mention when the specific one is already clear from
shared context — you and the reader both know which one because of the situation:

- "I'll fix **the** bug we discussed." (*the* bug — we both know which one, we
  discussed it.)
- "Can you check **the** deploy?" (In a team mid-deploy, *the* deploy — the one
  happening now; there's only one relevant one.)
- "**The** production server is down." (There's one production server we all mean.)

### *a* vs *an*: by sound, not spelling

Use **an** before a *vowel sound*, **a** before a *consonant sound* — and it's the
sound that matters, not the letter:

- **an** error, **an** hour (silent h — vowel sound), **an** API (starts with
  "ay" sound), **an** SSD (starts with "es" sound)
- **a** user (starts with "yoo" — a consonant sound, despite the vowel letter),
  **a** URL ("you-are-el"), **a** one-time token ("wun" — consonant sound)

Read it aloud: if the next word starts with a vowel *sound*, use *an*.

{: .note }
> **The quick test in practice</br>**
> When you're unsure whether to write <em>a</em> or <em>the</em>, ask yourself:
> "If the reader replied 'which one?', would that be a reasonable question?" If yes
> — they don't know which one yet — use <strong>a/an</strong>. If "which one?"
> would be silly because it's obvious which one you mean — use <strong>the</strong>.
> "I found a bug" → "which one?" is reasonable (you haven't said) → <em>a</em>. "I
> fixed the bug from this morning" → "which one?" is silly (you just said which) →
> <em>the</em>.

## Phrase Bank

Common work phrases with the right article:

| Situation | Correct form |
|---|---|
| Introducing a problem | "I found **a** bug / hit **an** error / have **a** question" |
| Referring back to it | "**The** bug / **the** error only happens when…" |
| A shared, obvious thing | "**the** deploy, **the** build, **the** production server, **the** PR" |
| First-time task | "Let me open **a** ticket / write **a** doc / create **a** branch" |
| Vowel-sound words | "**an** error, **an** hour, **an** API, **an** update, **an** issue" |
| Consonant-sound (tricky) | "**a** user, **a** URL, **a** UI, **a** one-off" |

## Checkpoint

**Q1.** What single question decides between *a/an* and *the*, and how does it
work?

**Your answer:**

<details>
<summary>Show Answer</summary>
<br>
The question is: <strong>"Does the reader already know WHICH one I mean?"</strong>
(Or, equivalently: "If they replied 'which one?', would that be a reasonable
question?") If the reader does <em>not</em> yet know which specific one — because
you're introducing it for the first time, or it's just one of several — use
<strong>a/an</strong> (the indefinite article: "I found <em>a</em> bug" — new, the
reader doesn't know which). If the reader <em>does</em> know which specific one —
because you mentioned it before, or it's clear from shared context — use
<strong>the</strong> (the definite article: "I fixed <em>the</em> bug" — the one we
both know about). It's the new-vs-known distinction: <em>a/an</em> introduces (new
information, unspecified), <em>the</em> refers to something already established
(known, specified). The most reliable everyday pattern that follows from this:
first mention gets <em>a/an</em> (introducing), later mentions get <em>the</em>
(now known) — "I opened <em>a</em> PR; <em>the</em> PR fixes the login bug." And
<em>the</em> works on first mention too when context already makes it clear which
one ("the production server," "the deploy we discussed"). This one question won't
cover every edge case in English, but it correctly handles the large majority of
a/an-vs-the choices you'll make in workplace writing.
</details>

**Q2.** Which is correct and why: "an API" or "a API"? "a user" or "an user"?

**Your answer:**

<details>
<summary>Show Answer</summary>
<br>
"<strong>an API</strong>" and "<strong>a user</strong>" are correct — and the
reason is that a/an is chosen by <em>sound</em>, not spelling. Use <em>an</em>
before a vowel <em>sound</em>, <em>a</em> before a consonant <em>sound</em>.
"<strong>an API</strong>": even though it's spelled with a consonant letter (A is
a vowel, but "API" is said letter-by-letter starting with "ay-pee-eye"), the first
<em>sound</em> is the vowel "ay," so it takes <em>an</em> ("an ay-pee-eye"). Say it
aloud: "a API" sounds wrong because you'd have two vowel sounds colliding.
"<strong>a user</strong>": "user" is spelled with the vowel letter U, but it's
<em>pronounced</em> "yoo-zer" — starting with a "y" consonant sound — so it takes
<em>a</em> ("a yoo-zer"), not "an user" (which sounds wrong). The rule is about how
the word <em>sounds</em> when you say it, not the letter it's spelled with. Other
common work examples: "<em>an</em> hour" (silent h — starts with a vowel sound),
"<em>an</em> SSD / <em>an</em> S3 bucket" (letter names starting with "es" vowel
sound), but "<em>a</em> URL" ("you-are-el" — consonant "y" sound), "<em>a</em> UI"
("you-eye"), "<em>a</em> one-time password" ("wun" — consonant "w" sound). The
practical test: read the phrase aloud, and if the next word starts with a vowel
sound, use <em>an</em>; if a consonant sound, use <em>a</em>. Trust your ear over
the spelling.
</details>

## Homework

Look at ten of your recent Slack messages or PR descriptions. Find every *a*,
*an*, and *the* (and every place you might have dropped an article). For each, check
it with the one question: does the reader know which one? Mark any you'd now change.
Then notice: do you more often drop articles entirely (write "I found bug" instead
of "I found a bug"), or use the wrong one (a instead of the)? That pattern goes on
your checklist. Bonus: check every "a/an" is right by *sound*.

**Your answer:**

<details>
<summary>Show Answer</summary>
<br>
The exercise reveals your personal article pattern, which is usually one of two
things (and knowing which is yours is what matters). Pattern one: <strong>dropping
articles entirely</strong> — writing "I found bug in login flow" instead of "I
found a bug in the login flow" — which is extremely common for speakers of
languages that don't have articles (the article simply doesn't feel necessary, so
it gets omitted). If this is your pattern, the fix on your checklist is "did I
include the article?" — a presence check, before worrying about which one. Pattern
two: <strong>using the wrong article</strong> — writing "the bug" on first mention
when it should be "a bug," or vice versa — which is the new-vs-known choice being
made wrong. If this is your pattern, the checklist item is the one question ("does
the reader know which one?"). Most people lean toward one pattern, and targeting
<em>your</em> pattern is far more efficient than trying to master all of article
usage at once. The article system in English is genuinely complex (there are edge
cases even native speakers do by feel), but the two high-frequency errors —
dropping articles, and the a/the confusion on first-vs-later mention — cover the
large majority of what actually goes wrong in workplace writing, and both are
handled by simple checks (presence, and the "which one?" question). The bonus (a/an
by sound) catches the smaller but noticeable "a API / an user" errors, which are
easy to fix once you know to read it aloud. Add whichever pattern is yours to your
growing checklist (Lesson 9). The reassuring meta-point: you don't need to master
the entire theory of English articles — you need to catch <em>your</em> specific
recurring article error, which is usually just one or two patterns, with a quick
check. Articles will never be effortless (they're hard), but a targeted check for
your pattern makes your writing noticeably more accurate, and the "which one?"
question plus a presence-check handle most cases. Next lesson covers the third
article situation — when to use <em>no</em> article at all — which is the other big
source of article errors (and the "a feedback / the production" mistakes).
</details>
